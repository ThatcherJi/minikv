# minikv-lite 架构笔记

本文档解释了 `minikv-lite` 项目里的工程实践与设计取舍。建议您对照源码进行阅读。

## 核心组件

`minikv-lite` 共有三个运行二进制进程：

- `coord`: 储存中心以及路由状态管理协调器。
- `volume`: 在各个节点真正落盘的本地化进程。
- `cli`: 用于管理和排查问题的简易客户端。

库结构的划分也非常直观：

- `src/store.rs`: 实现了 append-only（仅追加写入）的本地核心存储逻辑。
- `src/volume.rs`: 将 HTTP API 包裹在一层 `Store` 上，做隔离操作。
- `src/coord.rs`: 实现了服务注册表，一致性哈希计算，副本分发写同步与后台检查等。
- `src/ring.rs`: 分发层的一致性哈希环计算引擎，内置虚拟节点策略。

## 存储引擎模型

每个 volume 接管一个数据目录，并且往里面唯一的一个 `data.log` 文件写入。所有的对象记录都是 Append-only 的。在内存中的哈希索引用来指向 `data.log` 对应键值的最新有效偏移量。

格式布局规定：

```text
crc32: u32
seq: u64
op: u8
key_len: u32
value_len: u32
key bytes
value bytes
```

`crc32` 段覆盖该段之后的所有字段，以此在服务意外宕机后的文件启动回溯机制（Replay）中保护数据的完整性。如果在日志尾部出现了损坏段，文件可以直接将有损位置阶段，提供了一个最稳妥的数据找回场景。

写操作 (`put`) 抛送一个 `OP_PUT`，之后将指针引到最新的偏移量。删除操作抛送一个 `OP_DEL` 的墓碑节点并在内存表里把 key 剥离。不存在数据的删除操作会被作为无效调用驳回。

## 数据压缩设计 (Compaction)

由于模型简单，在单机节点的日志紧缩清理通过纯停顿 (Stop-the-world) 方式：获取内部数据对象的全量锁，只挑拣出还在活跃的非过期项写往 `data.compact`，确保安全刷到磁盘后，原位替换原本的 `data.log`并且把偏移量同步更新过来。这免除了引入读写在并发下的双锁或者影子文件的麻烦，代码虽然很简陋但却拥有非常安全且易重现的心智模型。

## 生命周期和集群治理

每个启动的 volume 都会向 coordinator 注册，并且定时保持心跳机制。如果在 `dead-after-secs` 所规划的时期内没心跳过来，协调器会将其踢下线，在请求路由的过程中它就会不再参与哈希环的分发选点，直到下一轮苏醒重新激活。但后台治理接口依然可以探测到它的历史尸体状态。

对于请求的分发：每次发起调用时都会根据主键的哈希从环上拾取出 N 个不重叠的实例节点作为副本落脚点。当写入的时候，实现了极其严格的 Fan-out 发送，必须 `write_quorum` 回复满意才算成功落库（教学版里的预设就是必须 N 个副本全部到位）。读取时则由于冗杂性的要求放宽了，按顺序只要其中一个存活的节点有该值，则优先抢跑上报该结果。

项目有意略去了复杂但容易让初学者迷失方向的细节（如：Quorum读取弥合、自动跨节点重分布的 Repair 机制，或是多配置协调选举集群的高可用）。这完全是站在更明确“易读易拆”的角度设计的教学范本。
